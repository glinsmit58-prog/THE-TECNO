# Workflow — قواعد العمل على TecnoGems

## 🎯 مبدأ أساسي

**لا تبدأ العمل قبل قراءة `project-context.md`** في هذا المجلد.
هناك تجد: الحالة الحالية، البنود المُنجزة، والبند التالي.

## 📦 Git Workflow

### 1. فرع جديد لكل PR

صيغة أسماء الفروع:

| النوع | الصيغة | مثال |
|------|--------|------|
| ميزة | `feat/<kebab-name>` | `feat/2fa-admin` |
| إصلاح | `fix/<kebab-name>` | `fix/csrf-api-guard` |
| أمان | `security/<vXX-topic>` | `security/v52-audit-table` |
| توثيق | `docs/<kebab-name>` | `docs/steering-setup` |
| Refactor | `refactor/<kebab-name>` | `refactor/blueprint-split` |
| CI/بنية | `chore/<kebab-name>` | `chore/ci-github-actions` |

### 2. Commit Messages

استخدم [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <summary>

<body-optional-with-bullet-points>
```

أمثلة صحيحة من تاريخ المشروع:
- `security(v50): fix 14 critical/high findings from audit`
- `security(v50.2): fix 22 medium/low findings from audit`

### 3. عنوان الـ PR

نفس صيغة الـ commit الرئيسي. يفضَّل **عربي** في الوصف + **إنجليزي** في العنوان.

### 4. وصف الـ PR (إلزامي)

كل PR يجب أن يحوي:

```markdown
## الغرض
(جملتان تشرحان ماذا ولماذا)

## التغييرات
(قائمة بالملفات + السطور تقريباً)

## الاختبار
(كيف اختُبر)

## Breaking changes
(لا / نعم — مع التفاصيل)

## متغيرات بيئة جديدة
(إن وجدت)
```

## 🔖 تعليمات الوسم داخل الكود

عند إضافة إصلاح أمني/معماري، أضف تعليق مرجعي:

```python
# V51 FEATURE (2FA): enforce TOTP on admin routes
# V52 SECURITY (A1): ...
```

هذا يسمح بـ `grep "V51"` لاحقاً لإيجاد كل التعديلات ذات الصلة.

## 🧪 الاختبار قبل الـ PR

- ✅ `python -c "import ast; ast.parse(open('app.py').read())"` لا يرفع استثناء
- ✅ المشروع يقلع بـ `FLASK_ENV=development python app.py`
- ✅ لا تعارض (conflicts) مع `main`
- ✅ **لا تُدخل secrets** في الكود أو `.env.example`

## 📝 بعد كل PR — تحديث الـ steering

**إلزامي ضمن نفس الـ PR:**

1. في `project-context.md`:
   - انقل البند من قسم «المتبقية» إلى قسم «المُنجزة» (مع رقم PR + رابطه)
   - حدّث «آخر تحديث» (commit + تاريخ + ملاحظة ماذا تم)
   - أضف أي قرار معماري جديد إلى قسم «القرارات المُتخذة»

2. إذا تغيَّر هيكل الملفات (بند A مثلاً):
   - حدّث قسم «البنية على القرص»

3. إذا تغيّر متغيِّر بيئة:
   - حدّث قسم «متغيرات البيئة المهمة»

## 🚫 ممنوعات

- ❌ **لا تُعدِّل** `main` مباشرة
- ❌ **لا force-push** على `main` أو فروع مدموجة
- ❌ **لا تحذف** ملفات `V*.md` القديمة (تاريخ المشروع)
- ❌ **لا تُضيف** اعتماديات بدون تبرير في وصف الـ PR
- ❌ **لا تُغيِّر** بنية `.env.example` بدون ذكر ذلك صراحةً
- ❌ **لا تُنشئ** ملفات اختبارات لم يطلبها المستخدم

## 💬 عند بدء محادثة جديدة

**الرسالة المتوقَّعة من المستخدم:**

- "تابع من حيث توقفنا" → اقرأ steering ونفّذ البند التالي حسب الأولوية
- "نفّذ [اسم بند]" → اقرأ steering ثم ابدأ بذاك البند
- أي مهمة أخرى → اقرأ steering للسياق ثم نفّذ المطلوب

**الرد الأول منك يجب أن يتضمّن:**

1. تلخيص مختصر للحالة الحالية (2-3 أسطر من steering)
2. ما الذي ستفعله الآن
3. خطة تنفيذ مختصرة

ثم تبدأ العمل مباشرة — لا تسأل المستخدم أسئلة يجيب عنها الـ steering.

## 🎨 أسلوب الرد مع المستخدم

- اللغة: **عربي** في الرد العام، **إنجليزي** في الكود والـ commit messages
- الأسلوب: مباشر وموجَز، قوائم وجداول عند الحاجة
- لا تُسهب في شرح ما يعرفه المستخدم
- عند الأخطاء: اذكر السبب المحتمل ثم الحل، لا "آسف"
