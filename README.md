# موقع متجر شحن الألعاب

موقع مستقل عن تيليجرام، بواجهة مستخدم ولوحة إدارة في نفس المشروع.

## المميزات

- تسجيل مستخدم عادي بالبريد وكلمة المرور.
- فصل الصلاحيات: user / admin.
- سيرفر 1 وسيرفر 2 بدون ذكر أسماء المصادر للمستخدم.
- محفظة داخلية.
- طلبات تنفيذ بالخلفية حتى لا تتجمد الواجهة.
- لوحة إدارة:
  - إحصائيات
  - الطلبات
  - المستخدمون
  - تعديل الأرصدة
  - أرصدة السيرفرات

## التشغيل المحلي

1. ثبّت المتطلبات:

```bash
pip install -r requirements.txt
```

2. انسخ ملف الإعدادات:

```bash
copy .env.example .env
```

3. عدّل `.env` وضع مفاتيح السيرفرات.

4. شغّل مزامنة المنتجات:

```bash
python sync_products.py
```

5. شغّل الموقع:

```bash
python app.py
```

ثم افتح:

```text
http://127.0.0.1:5000
```

## حساب الأدمن الافتراضي

يتم إنشاؤه من `.env`:

```text
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=admin123456
```

غيّرها قبل التشغيل الحقيقي.

## نظام شحن الرصيد

تم إضافة نظام شحن يدوي كامل:

### للمستخدم
- يدخل إلى المحفظة.
- يختار طريقة الدفع.
- يرى العنوان/الرقم والتعليمات.
- يدخل المبلغ.
- يرسل إثبات الدفع أو رقم العملية.
- ينتظر موافقة الإدارة.

### للأدمن
من لوحة الإدارة:
- `طلبات الشحن`: قبول أو رفض طلبات الشحن.
- `طرق الدفع`: تعديل الاسم، الأيقونة، العنوان، التعليمات، التفعيل/التعطيل.
- `الإعدادات`: تعديل حساب أو رابط الدعم.

عند قبول طلب الشحن تتم إضافة الرصيد للمستخدم تلقائيًا.


## تحديث احترافي جديد

### طرق الدفع والعملات
- USDT و Binance وشام كاش دولار: بالدولار.
- شام كاش سوري، سيرياتيل كاش، ضمن المركز: بالليرة السورية.
- يمكن تعديل عملة أي طريقة دفع من لوحة الإدارة.
- سعر الصرف موجود في: لوحة الإدارة > الإعدادات.

### شحن الرصيد
- المستخدم يختار طريقة الدفع.
- تظهر العناوين والتعليمات بوضوح.
- يستطيع رفع صورة إيصال الدفع أو كتابة رقم العملية.
- إذا دفع بالليرة السورية، يتم تحويلها إلى رصيد دولار حسب سعر الصرف.

### آلية الطلبات
- عند شراء المستخدم، يصبح الطلب قيد التنفيذ.
- إذا كان وضع الطلبات اليدوية مفعلًا، يتحول الطلب إلى "تنفيذ يدوي" ولا يُرسل للمورد.
- إذا كان الوضع التلقائي مفعلًا:
  - يحاول النظام إرسال الطلب للمورد.
  - إذا نجح: يصبح مكتملًا.
  - إذا فشل: يتحول إلى "تنفيذ يدوي" مع ظهور سبب الفشل للأدمن.
- قبول الطلب يدويًا من الإدارة يعني أنك نفذته فعليًا خارج النظام، وليس تنفيذًا وهميًا تلقائيًا.


## إصلاح السيرفر 1 / G2Bulk

تم تعديل الربط مع G2Bulk ليستخدم الصيغة الصحيحة التي تم اختبارها في Postman:

```text
POST https://api.g2bulk.com/api/v2
Body: x-www-form-urlencoded

key       = G2BULK_API_KEY
action    = add
service   = SERVICE_ID
link      = Player ID
quantity  = 1
```

كما تم تعديل `sync_products.py` ليحاول جلب خدمات السيرفر 1 مباشرة من API عبر `action=services`.
إذا فشل الجلب من API، يستخدم ملف `g2bulk_services.json` كاحتياط.

## إظهار اسم اللاعب قبل الشحن

تم إضافة ميزة التحقق من اللاعب قبل الشراء:

- في صفحة الشراء، عند كتابة Player ID يظهر اسم اللاعب إن كان المزود يدعم التحقق.
- يمكن تعطيل أو تفعيل الميزة من:
  لوحة الإدارة > الإعدادات > تفعيل إظهار اسم اللاعب قبل الشحن
- Shop2Topup يستخدم endpoint:
  `/player/validate`
- G2Bulk قد يختلف حسب الحساب/الخدمة، لذلك يجرب عدة أكشنات تحقق، وإذا لم تكن مدعومة يسمح للمستخدم بالمتابعة مع رسالة تنبيه بدل تعطيل الشراء.


## تحسين سجل الطلبات والتحقق

- يظهر Player ID في سجل طلبات المستخدم.
- يظهر Player ID في سجل طلبات الإدارة.
- تاريخ الطلبات يظهر بتوقيت سوريا UTC+3.
- إذا تحقق Shop2Topup من اللاعب دون إرجاع الاسم، تظهر رسالة: تم التحقق بنجاح لكن المزود لم يرجع الاسم.
- إذا لم يدعم G2Bulk إظهار اسم اللاعب، تظهر رسالة تنبيه ولا يتم منع الشراء.


## تحديث الطلبات والسيرفرات والتصميم

- تمت إزالة خاصية إظهار اسم اللاعب قبل الشحن.
- عند إنشاء الطلب تكون الحالة أولاً: `بانتظار التنفيذ`.
- عندما يبدأ النظام بإرسال الطلب للمورد تصبح الحالة: `قيد التنفيذ`.
- إذا نجح المورد تصبح الحالة: `مكتمل`.
- إذا فشل المورد أو كان الوضع اليدوي مفعلًا تصبح الحالة: `بانتظار تنفيذ يدوي`.
- يمكن من لوحة الإدارة > الإعدادات إخفاء أو إظهار سيرفر 1 وسيرفر 2.
- تم تحسين عرض الألعاب وإضافة صور صغيرة جذابة للألعاب.


## تحديث حالة الطلب تلقائياً كل 5 ثوانٍ

تم تعديل آلية الطلبات:

- عند إنشاء الطلب: `بانتظار التنفيذ`.
- عند بدء الإرسال للمورد: `جاري التنفيذ لدى المورد`.
- إذا قبل المورد الطلب لكنه ما زال Pending/Processing: `قيد التنفيذ لدى المورد`.
- لا يصبح الطلب `مكتمل` إلا إذا رجع المورد بحالة `completed`.
- يوجد فحص خلفي تلقائي كل 5 ثوانٍ يراجع حالة الطلب لدى المورد.
- صفحات المستخدم والإدارة تحدث الحالة بصريًا كل 5 ثوانٍ دون إعادة تحميل الصفحة.

مهم:
- يجب أن يرجع المورد رقم طلب صالح في `provider_order_id`.
- Shop2Topup يتم فحصه عبر `GET /orders/:orderId`.
- G2Bulk يتم فحصه عبر `action=status` مع رقم الطلب.

## إرجاع آلية حالة الطلب

## تفعيل البريد الإلكتروني

تمت إضافة نظام تفعيل البريد الإلكتروني:

- يمكن تفعيله أو تعطيله من لوحة الإدارة > الإعدادات.
- عند التفعيل، المستخدم الجديد يستلم رابط تفعيل على بريده.
- لا يستطيع المستخدم تسجيل الدخول قبل تفعيل البريد.
- الأدمن لا يحتاج تفعيل بريد.
- إعدادات SMTP تكون في ملف `.env`.

### إعدادات `.env` المطلوبة

```env
BASE_URL=https://tecnogems.com
MAIL_SERVER=smtp.gmail.com
MAIL_PORT=587
MAIL_USE_TLS=1
MAIL_USERNAME=your-email@gmail.com
MAIL_PASSWORD=your-app-password
MAIL_FROM=your-email@gmail.com
```

مهم: إذا رفعت هذه النسخة فوق السيرفر، لا تستبدل ملف `data/site.db` الحالي حتى لا تضيع بيانات المستخدمين والطلبات.

## إصلاح قراءة مفاتيح API وتجهيز الألعاب

هذه النسخة تصلح مشكلة قراءة مفاتيح الموردين من `.env` داخل `providers.py`.

كما تقوم تلقائيًا عند أول تشغيل بتجهيز كتالوج المورد 1 من ملف `g2bulk_services.json` داخل لوحة الإدارة:
- الألعاب المشهورة مثل Free Fire / PUBG / FC تكون مفعلة.
- باقي التصنيفات تظهر في الإدارة غير مفعلة، وتستطيع تفعيل ما تريد.

بعد رفع النسخة أعد تشغيل الخدمة، ثم ادخل: الإدارة > الألعاب.

لجلب باقات المورد 2 بعد وضع المفتاح الحقيقي:

```bash
cd /root/project
python3 sync_products.py
# أو تجربة فئات كثيرة من المورد 2
SYNC_SERVER2_RANGE=1 SYNC_SERVER2_START=1 SYNC_SERVER2_END=80 python3 sync_products.py
systemctl restart game-topup
```

مهم: لا تضع `Bearer` داخل `SHOP2TOPUP_API_KEY`؛ ضع المفتاح فقط.

## Gmail BadCredentials
إذا ظهر الخطأ 535 Username and Password not accepted، فالمشكلة من إعدادات Gmail وليست من الكود. استخدم App Password جديد من Google، وضعه في MAIL_PASSWORD. يمكن لصقه مع أو بدون مسافات، والكود سيزيل المسافات تلقائيًا. تأكد أن MAIL_USERNAME و MAIL_FROM لنفس البريد.

## إخفاء الباقات
في صفحة ترتيب الباقات: إذا وضعت sort_order = 0 للباقة فلن تظهر في واجهة المستخدم، بشرط وجود باقات أخرى بترتيب أكبر من 0.
